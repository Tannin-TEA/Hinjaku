# Gemini-MEMO: 開発作業用スクラッチパッド

## 【現在進行中のタスク】
PHASE 1: 構造分解（Structural Decomposition）

## 【ユーザーからの命令】
1. 挙動を変えずにリファクタリングすること。
2. 以下の3層に分離すること：
   - 入出力層（CLI / ファイル / 外部依存）
   - コアロジック層（純粋関数）
   - データ構造層
3. main関数は「呼び出しだけ」にする。
4. 各関数の責務を1つに限定する。
5. 依存関係は「外 → 内」の一方向に限定する。

## 【Geminiの提案：モジュール構成案】
### 1. データ構造層 (Data Layer)
- `config.rs`: `Config` 構造体、`SortMode` 等の定義。
- `archive.rs`: `ImageEntry` 構造体の定義。

### 2. コアロジック層 (Core Logic Layer)
- `archive.rs`: 判定ロジック（`is_image_ext`, `natord` 等）。
- `manager.rs`: 管理ロジック（`schedule_prefetch`, `apply_sorting` 等）。
- ※OSやUIに依存しない「純粋な計算」をここに集約。

### 3. 入出力層 (I/O Layer)
- `main.rs`: アプリの起動、引数受取、他モジュールの呼び出し。
- `integrator.rs` (新規): Windows固有処理（二重起動防止、ShellExecuteW等）。
- `config.rs`: INIファイルの物理的な読み書き。
- `viewer.rs`: eguiによるUI描画とイベント受付。

## 【決定事項（実行待ち）】
1. **プロファイル管理**: `hinjaku.exe -c sub.ini` のように起動することで、設定を完全に分離する。
2. **INI構造**:
   - `[Global]`: 共通動作（ソート、マンガモード、多重起動防止等）
   - `[App_...]`: 外部連携（実行パス、引数テンプレート）
   - `[KeyConfig]`: キー割り当て（将来の実装に向けた予約）
3. **ライブラリ**: `serde`, `toml`, `chrono` を完全に削除。
4. **モジュール構成**:
   - `main.rs`: 起動のみ。
   - `integrator.rs`: Windows連携、引数パース、アイコン生成。
   - `config.rs`: INI読み書き、プロファイル指定対応。
   - `viewer.rs` / `archive.rs` / `manager.rs`: 分離後の新関数呼び出しへ修正。
