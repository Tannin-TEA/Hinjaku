# Windows API 準拠によるパス通信と管理の再定義

## 1. 現状の課題
これまでの「事後的に `\\?\` を除去する」という手法は、根本解決になっておらず設計ミスであった。
Windows APIとの通信において `\\?\` が混入することを前提とせず、APIが期待する標準的なパス形式を最初から提供する必要がある。

## 2. 基本指針：Rust Native & Windows API Integration

- **内部処理**:
  Rustの標準的な流儀に従い、パスの保持や操作には `PathBuf` を使用する。

- **通信（IPC）**:
  二重起動時のパス転送には、独自の通信機構（Rustのチャンネル等のラップ）を排除し、Windows標準の `WM_COPYDATA` メッセージ通信のみを使用する。

- **外部連携**:
  外部アプリの起動には `ShellExecuteW` を使用する。引数として渡すパス文字列は、Windows APIがネイティブに解釈可能な「非UNC形式」のUTF-16文字列として構築する。

## 3. 禁止事項
- **置換による回避の禁止**: `replace(r"\\?\", "")` などのその場しのぎの文字列操作は行わない。
- **独自IPCの禁止**: Windows環境においてOSが提供するメッセージング以外の通信手段を混ぜない。

## 4. 実行手順
1. 入力（CLI/D&D）時にパスを正規化し `PathBuf` に格納する。
2. Windows API呼出直前で、OSが解釈可能なUTF-16形式へ直接エンコードし、APIの引数に流し込む。
